Debugging ROS 2 Systems

Introduction​

Your code breaks. That's not failure—that's learning. In this lesson, you'll learn how to fix it.

We'll start with the diagnostic toolkit—command-line tools that let you see inside a running ROS 2 system. Then, you'll debug broken code while collaborating with AI.

Time estimate: 60 minutes

The Diagnostic Toolkit​

When something doesn't work, you ask: "What's running, and what data is flowing?"

The answer comes from four commands:

  1. ros2 node list — What nodes exist?
  2. ros2 node info — What does a node publish/subscribe?
  3. ros2 topic list — What topics are active?
  4. ros2 topic echo — What data is flowing through a topic?
  5. Logging — What is each node doing?

Command 1: ros2 node list​

What it does: Shows all running nodes

ros2 node list

Example output:

/turtle_command_publisher
/turtlesim

What it tells you:

  • Your publisher node is running ✅
  • Turtlesim is running ✅

What it doesn't tell you:

  • If nodes are working correctly (just that they exist)
  • If they're communicating

Use case: "Is my node actually running?"

Command 2: ros2 node info​

What it does: Shows what a node publishes and subscribes to

ros2 node info /turtle_command_publisher

Example output:

Node /turtle_command_publisher
  Publishers:
    /turtle1/cmd_vel: geometry_msgs/msg/Twist
  Subscribers:
    (none)
  Service Servers:
    (none)
  Service Clients:
    (none)
  Action Servers:
    (none)
  Action Clients:
    (none)

What it tells you:

  • Your publisher publishes to /turtle1/cmd_vel ✅
  • It sends Twist messages ✅
  • It doesn't subscribe to anything (correct for a publisher)

Use case: "What is my node actually connected to?"

Command 3: ros2 topic list​

What it does: Shows all active topics

ros2 topic list

Example output:

/turtle1/cmd_vel
/turtle1/color_sensor
/turtle1/pose
/turtlesim/describe_parameters
...

What it tells you:

  • Topics that exist in the system
  • If your topic (/turtle1/cmd_vel) is being created

Use case: "Does the topic exist? Is it spelled correctly?"

Command 4: ros2 topic info​

What it does: Shows details about a topic

ros2 topic info /turtle1/cmd_vel

Example output:

Type: geometry_msgs/msg/Twist
Publisher count: 1
Subscriber count: 1

What it tells you:

  • Type: The message type (Twist) is correct
  • Publisher count: 1 node is publishing (your publisher)
  • Subscriber count: 1 node is subscribing (Turtlesim)

What it should be:

  • Exactly 1 publisher (your node)
  • At least 1 subscriber (Turtlesim)

Use case: "Are the right nodes connected to this topic?"

Command 5: ros2 topic echo​

What it does: Shows actual messages flowing through a topic in real-time

ros2 topic echo /turtle1/cmd_vel

Example output (continuous):

linear:
  x: 1.0
  y: 0.0
  z: 0.0
angular:
  x: 0.0
  y: 0.0
  z: 0.0
---
linear:
  x: 1.0
  y: 0.0
  z: 0.0
angular:
  x: 0.0
  y: 0.0
  z: 0.0
---

What it tells you:

  • Messages ARE being published ✅
  • The values are correct (linear.x = 1.0) ✅
  • Updates happen every 0.5 seconds (matching your timer) ✅

What it doesn't tell you:

  • If the subscriber is actually processing the messages (you need to watch the Turtlesim window for that)

Use case: "Are messages actually flowing? Are they the right values?"

Logging: What the Node Is Doing​

Your code includes logging statements:

self.get_logger().info(f'Publishing: linear.x={msg.linear.x}')

When you run the node, you see:

[INFO] [turtle_command_publisher]: Publishing: linear.x=1.0
[INFO] [turtle_command_publisher]: Publishing: linear.x=1.0

This tells you:

  • The code reached this line ✅
  • The value is correct ✅
  • The timing is right (5 messages in ~2.5 seconds) ✅

Add logging to debug:

def timer_callback(self):
    self.get_logger().info(f'Timer fired! Counter={self.counter}')  # Add this

    msg = Twist()
    msg.linear.x = 1.0

    self.get_logger().info(f'About to publish: {msg}')  # Add this
    self.publisher_.publish(msg)
    self.get_logger().info('Published successfully')    # Add this

    self.counter += 1

Now when you run the node, you see exactly what's happening step-by-step.

Systematic Debugging: The 5-Question Method​

When something doesn't work, ask these questions in order:

Question 1: Is the Node Running?​

ros2 node list | grep your_node_name

If you see your node, ✅ it's running. If not, the node crashed or didn't start.

Next step: Check terminal output for error messages.

Question 2: Is the Topic Created?​

ros2 topic list | grep /your/topic

If you see the topic, ✅ the node created it. If not, the create_publisher() failed (check import statement).

Next step: Check for Python syntax errors.

Question 3: Is the Topic Receiving Messages?​

ros2 topic echo /your/topic

If you see messages appearing, ✅ the publisher is working. If nothing appears, the callback is never called (check timer).

Next step: Check timer creation and callback naming.

Question 4: Are Messages Correct?​

Look at the output from ros2 topic echo. Are the values what you expect?

If values are correct, ✅ your message creation is right. If values are wrong, check msg.field = value assignments.

Next step: Verify message field names and types.

Question 5: Is the Subscriber Responding?​

Look at the Turtlesim window (or other subscriber behavior).

If the turtle moves, ✅ the subscriber is receiving and processing. If nothing happens, check topic name and message type match.

Next step: Use ros2 topic info to verify subscribers exist.

Common Errors & Diagnosis​

Error: "ros2: command not found"​

Diagnosis: ROS 2 not in your shell environment

Command to check:

source /opt/ros/humble/setup.bash

Then try again.

Error: "Node '/my_node' not found"​

Diagnosis: Node crashed before you could query it, or name is different

Commands to check:

ros2 node list  # See what nodes ARE running
# Check terminal where node was running for error messages

Error: Topic doesn't appear in ros2 topic list​

Diagnosis: create_publisher() line has an error

Typical causes:

  • Topic name wrong (spelling, capitalization)
  • Message type doesn't exist (missing from X.msg import Y)
  • Import statement wrong

Fix: Check imports and create_publisher() call

Error: Messages don't appear in ros2 topic echo​

Diagnosis: Publisher created, but callback never fires

Typical causes:

  • Timer not created
  • Timer callback name spelled wrong
  • rclpy.spin() not called

Fix: Check:

# In __init__:
self.timer = self.create_timer(0.5, self.timer_callback)

# In main:
rclpy.spin(node)

Error: Messages appear but are all zeros​

Diagnosis: Callback is publishing, but message fields not set

Typical cause:

msg = Twist()  # Default: all zeros
# Forgot to set fields!
self.publisher_.publish(msg)

Fix: Set message fields:

msg = Twist()
msg.linear.x = 1.0  # This line was missing
self.publisher_.publish(msg)

Error: Subscriber isn't responding (topic shows messages, but nothing happens)​

Diagnosis: Topic/message type mismatch

Commands to check:

ros2 topic info /turtle1/cmd_vel  # Check message type
ros2 node info /turtlesim         # Check what type subscriber expects

Typical cause: You're publishing to /turtle1/cmd_vel but subscriber expects different message type

Fix: Use ros2 topic info to see expected type, match your publisher

🤝 Collaboration Exercise: Debug This System​

I've created intentionally broken code. Your job: diagnose and fix it using the toolkit.

Setup​

In Terminal 1, run Turtlesim:

ros2 run turtlesim turtlesim_node

In Terminal 2, save this code as broken_publisher.py:

import rclpy
from rclpy.node import Node
from geometry_msgs.msg import Twist

class BrokenPublisher(Node):
    def __init__(self):
        super().__init__('broken_publisher')

        # BUG 1: Wrong topic name
        self.publisher_ = self.create_publisher(Twist, '/turtle1/move_cmd', 10)

        self.timer = self.create_timer(0.5, self.timer_callback)
        self.counter = 0

    def timer_callback(self):
        msg = Twist()
        # BUG 2: Message values are wrong
        msg.linear.x = 0.0  # Supposed to move forward, but this is 0!
        msg.angular.z = 0.0

        self.publisher_.publish(msg)
        self.counter += 1

        if self.counter >= 5:
            rclpy.shutdown()

def main(args=None):
    rclpy.init(args=args)
    node = BrokenPublisher()
    rclpy.spin(node)

if __name__ == '__main__':
    main()

In Terminal 2, run the broken code:

python3 broken_publisher.py

In Terminal 1 (Turtlesim window): Nothing happens (turtle doesn't move)

Your Diagnosis Task​

Use the 5-question method to find the bugs:

Question 1: Is the node running?

ros2 node list | grep broken

Question 2: Is the topic created?

ros2 topic list
# Do you see /turtle1/move_cmd? Or /turtle1/cmd_vel?

Question 3: Are messages being published?

ros2 topic echo /turtle1/move_cmd  # or whatever topic you found

Question 4: Are messages correct?

# Look at the echo output. What's the linear.x value?

Question 5: Is the subscriber responding?

# Does the turtle move? Check Turtlesim window
ros2 node info /turtlesim  # What topics does it subscribe to?

Work With AI to Fix It​

Once you've diagnosed, ask Claude:

"I'm publishing to /turtle1/move_cmd but Turtlesim subscribes to /turtle1/cmd_vel. Also, my message has linear.x = 0.0 so the turtle doesn't move. How do I fix this?"

Claude will help you understand:

  1. Topic names must match exactly
  2. Message values must be non-zero to have effect
  3. The subscriber doesn't care who publishes—it just reads the topic

Fix the Code​

Modify broken_publisher.py:

# Fix BUG 1: Change topic name
self.publisher_ = self.create_publisher(Twist, '/turtle1/cmd_vel', 10)

# Fix BUG 2: Change message value
msg.linear.x = 1.0  # Now it will move!

Run again. The turtle should now move forward.

Self-Assessment​

Before moving to Lesson 5, verify:

  • Can use ros2 node list to see running nodes
  • Can use ros2 node info to see what node publishes/subscribes
  • Can use ros2 topic list to see active topics
  • Can use ros2 topic info to see message type and subscriber count
  • Can use ros2 topic echo to watch messages flow
  • Understood the 5-question debugging method
  • Successfully diagnosed and fixed the broken code
  • Collaborated with AI to understand the fixes

If all checkboxes pass, you're ready for Lesson 5: Your First Subscriber.
