What is ROS 2?

Introduction​

Before writing any code, let's understand why ROS 2 exists and how it thinks about robot systems.

You've probably built simple programs before: a script that reads a file, does some math, writes output. One process. One thread. Simple.

But robots are different. A robot doesn't do one thing. It does many things simultaneously:

  • Read sensor data from a camera (continuously)
  • Process that data (detect obstacles)
  • Make a decision (turn left or right)
  • Send commands to motors (move the wheel)
  • Update a map (remember where we've been)
  • All of this happening at the same time.

Traditional single-process programming breaks here. You can't both "read camera" and "move motor" in one sequence. They need to happen independently.

That's where ROS 2 comes in. ROS 2 lets you build robots as a collection of small, independent processes that talk to each other.

In this lesson, you'll learn the fundamental architecture: nodes, topics, and the pub/sub pattern. No code yet—just understanding.

Why Does ROS 2 Exist?​

The Problem: Robot Systems Are Complex​

Imagine a mobile robot that needs to:

  1. Read Lidar sensor data (streaming, 30 Hz)
  2. Process the Lidar to build a map
  3. Detect obstacles
  4. Plan a path
  5. Send motion commands to wheels
  6. Monitor battery voltage
  7. Send status updates to a monitoring computer

If you try to do all this in one program:

# ❌ BAD: Everything in one process
while True:
    lidar_data = sensor.read()        # Blocks here waiting for sensor
    map_data = process(lidar_data)    # Slow processing
    obstacles = detect(map_data)      # More processing
    plan = plan_path(obstacles)       # More processing
    motors.move(plan)                 # Finally move motors
    battery = monitor_battery()       # Check battery
    # By the time we loop back, Lidar has 100 unread messages!

Problem: Reading the Lidar blocks until the sensor sends data. But what if the sensor is slow? The motors wait. Processing takes 500 ms? The motors wait. Everything waits for everything else.

The Solution: Independent Processes​

What if instead, each task ran in its own process?

┌──────────────────┐
│  Lidar Reader    │  Continuously reads sensor (30 Hz)
│  (Process 1)     │
└────────┬─────────┘
         │ publishes Lidar data
         ↓
┌──────────────────┐
│  Map Builder     │  Processes Lidar independently
│  (Process 2)     │
└────────┬─────────┘
         │ publishes Map
         ↓
┌──────────────────┐
│  Obstacle        │  Detects obstacles independently
│  Detector        │
│  (Process 3)     │
└────────┬─────────┘
         │ publishes Obstacles
         ↓
┌──────────────────┐
│  Motion Planner  │  Plans path independently
│  (Process 4)     │
└────────┬─────────┘
         │ publishes Goal
         ↓
┌──────────────────┐
│  Motor Driver    │  Moves wheels independently
│  (Process 5)     │
└────────────────┘

Now:

  • Lidar Reader runs at 30 Hz even if Map Builder is slow
  • Motor Driver executes commands immediately, without waiting for planning
  • If one process crashes, others keep running
  • You can test each process independently
  • You can add new processes (e.g., "Battery Monitor") without changing others

This is the ROS 2 model.

Nodes: Independent Processes​

A node is an independent process that does one thing well.

What's a Node?​

Think of a node like a person on a robot team:

ConceptTeam AnalogyRobot System
PersonHas a job (driver, navigator, watchman)Node has a responsibility (read sensor, plan path, move motor)
CommunicationPeople talk to each otherNodes send messages to each other
IndependenceEach person works on their taskEach node runs its code independently
SpecializationDriver focuses on drivingPublisher node focuses on publishing data

Examples of ROS 2 Nodes​

Sensor Reader Node

  • Reads data from a physical sensor (camera, Lidar)
  • Sends that data to other nodes
  • Runs continuously or on a timer
  • Code: ~50 lines of Python

Image Processor Node

  • Subscribes to camera images
  • Processes them (detect faces, find objects)
  • Publishes processed results
  • Runs whenever an image arrives

Navigation Node

  • Reads sensor data (obstacles)
  • Reads robot position
  • Publishes commands: "move forward", "turn left"
  • Makes decisions about where to go

Motor Driver Node

  • Reads motion commands
  • Sends electrical signals to motors
  • Reports back: "I moved 1 meter"

Each node is separate, independent, and specialized.

Node Names​

Every node has a unique name, like:

  • /camera_reader
  • /obstacle_detector
  • /motor_controller
  • /battery_monitor

The / means "this is a global name, unique across the entire robot system".

Topics: Communication Channels​

A topic is a named communication channel. Nodes send data to topics. Other nodes read from topics.

The Bulletin Board Analogy​

Imagine a physical bulletin board in a robot office:

┌─────────────────────────────┐
│   BULLETIN BOARD: "/camera  │
│                             │
│   [Image from camera]       │
│   [Image from camera]       │
│   [Image from camera]       │
│                             │
└─────────────────────────────┘

Anyone can write to this board (post images). Anyone can read from it (take images). The writer doesn't need to know who reads. The reader doesn't need to know who writes.

Examples of Topics​

/camera/image
  → Camera node publishes new images here every 33 ms
  → Object detector subscribed, reads every new image
  → Display node subscribed, shows images on screen

/lidar/scan
  → Lidar node publishes scan data every 50 ms
  → Mapping node subscribed
  → Obstacle detector subscribed
  → Both read the same data independently

/cmd_vel
  → Planning node publishes motion commands
  → Motor driver subscribed, moves wheels
  → Autonomous override node might also read it

/battery/voltage
  → Battery monitor publishes voltage every second
  → Safety monitor subscribed, checks if voltage too low
  → Dashboard subscribed, displays battery level

Topic Naming Convention​

Topics use hierarchical names with forward slashes:

✅ Good topic names:

  • /robot_arm/joint1/state (clear hierarchy: robot → subsystem → specific data)
  • /camera/rgb/image (device → sensor type → data)
  • /lidar/scan (device → data type)
  • /cmd_vel (command velocity—standard ROS 2 name)

❌ Bad topic names:

  • /data (too vague, what data?)
  • /sensor_topic_data (redundant "topic" in name)
  • /LIDAR (inconsistent capitalization)
  • /scan (no hierarchy, ambiguous)

Publishers and Subscribers: The Communication Pattern​

Publisher: A node that sends data to a topic. Subscriber: A node that receives data from a topic.

How Pub/Sub Works​

Publisher (Camera Reader Node)
    │
    │ "Here's a new image"
    ↓
┌──────────────────────┐
│  Topic: /camera/image│
└──────────────────────┘
    ↑           ↑
    │           │
    │ reading   │ reading
    │           │
Subscriber 1      Subscriber 2
(Obstacle Detector) (Display)

Process:

  1. Camera Reader (publisher) captures an image
  2. Sends it to the topic /camera/image
  3. Obstacle Detector (subscriber) is listening to /camera/image
    • Automatically receives the image
    • Processes it
    • Publishes obstacles to /obstacles
  4. Display (another subscriber) is also listening to /camera/image
    • Receives the same image
    • Shows it on a screen
  5. Neither subscriber knows the other exists. Neither knows who publishes. They just read from the topic.

Why This Is Powerful​

Loose Coupling: Nodes don't depend on each other directly.

  • Add a new subscriber (e.g., "ML detector")? No changes needed to the publisher.
  • Replace the camera with a better one? No changes to subscribers.

Scalability: Multiple publishers or subscribers on the same topic.

  • 3 different nodes want Lidar data? All subscribe to /lidar/scan. The Lidar publisher doesn't care.

Flexibility: Easy to test in isolation.

  • Want to test the obstacle detector? Create a fake camera publisher.
  • Want to test the display? Play back recorded images.

The Pub/Sub Pattern in Action​

Let me show you a concrete example with a real robot scenario.

Scenario: Mobile Robot Obstacle Avoidance​

Goal: Robot moves forward, avoiding obstacles.

Nodes:

  1. Lidar Publisher — Publishes sensor scans
  2. Obstacle Detector — Reads scans, detects obstacles
  3. Path Planner — Reads obstacles, plans movement
  4. Motor Controller — Reads plan, moves wheels

Topics:

  • /lidar/scan — Raw sensor data
  • /obstacles/detected — Detected obstacles
  • /motion/goal — Target movement
  • /wheels/cmd_vel — Motor commands

Timeline (one cycle, ~100 ms):

t=0 ms:
  Lidar publishes to /lidar/scan

t=5 ms:
  Obstacle Detector reads /lidar/scan, processes it

t=20 ms:
  Obstacle Detector publishes to /obstacles/detected

t=25 ms:
  Path Planner reads /obstacles/detected, plans route

t=50 ms:
  Path Planner publishes to /motion/goal

t=55 ms:
  Motor Controller reads /motion/goal, sends signals to motors

t=100 ms:
  Lidar publishes again (repeats cycle)

Key insight: Each node processes at its own speed. Lidar doesn't wait for planning. Motors don't wait for detection. Everything happens in parallel.

System Architecture: Drawing Your First System​

Let's apply this thinking to design a simple robot system.

Exercise: Imagine a Robot​

Your robot:

  • Has a camera
  • Needs to detect people
  • Needs to move toward detected people

What nodes would you need?

  1. Camera Publisher — Reads camera, publishes images
  2. Person Detector — Reads images, detects people, publishes person locations
  3. Motion Planner — Reads person locations, computes "move toward person" commands
  4. Motor Controller — Reads commands, moves motors

What topics?

  • /camera/image — Raw camera images
  • /people/detected — Detected person locations
  • /motion/commands — Movement commands
  • /wheels/velocity — Motor velocity targets

Draw Your System​

On paper or in an image, draw:

  1. Boxes for each node (label with node name)
  2. Arrows between nodes through topics (label arrow with topic name)
  3. Mark which node is publisher and which is subscriber

Your diagram should look like:

┌────────────────┐
│ Camera Publisher│
└────────┬───────┘
         │ publishes /camera/image
         ↓
┌────────────────────┐
│ Person Detector    │
│ (subscriber)       │ publishes /people/detected
└────────┬───────────┘
         ↓
┌────────────────────┐
│ Motion Planner     │
│ (subscriber)       │ publishes /motion/commands
└────────┬───────────┘
         ↓
┌────────────────────┐
│ Motor Controller   │
│ (subscriber)       │
└────────────────────┘

Why ROS 2 Wins​

Now you understand why ROS 2 is so popular:

  1. Independence: Nodes run independently. Fast nodes don't wait for slow ones.

  2. Scalability: Add 10 nodes? Same architecture works. Replace one node? Others don't break.

  3. Reusability: A camera reader works the same way whether it's reading from a USB camera or IP camera.

  4. Testing: Test each node independently. Create fake publishers for testing.

  5. Team development: Different people write different nodes. Interfaces (topics) stay consistent.

  6. Debugging: Each node logs its own data. Easy to trace where problems happen.

  7. Standard: ROS 2 is used in industry (robotics companies), research (universities), and open-source (lots of community code you can reuse).
