Your First Subscriber

Introduction​

In Lesson 3, you built a publisher—a node that sends messages. Now you'll build a subscriber—a node that receives messages.

A subscriber doesn't run on a timer (like a publisher often does). Instead, it waits for messages and processes them when they arrive. This is called event-driven programming.

What you'll build:

class TurtlePoseReader(Node):
    # Subscribes to /turtle1/pose
    # When Turtlesim sends position data, callback function fires
    # Prints position data to terminal

Understanding Subscribers: The Theory​

Subscriber Role​

A subscriber is a node that listens to a topic and processes messages when they arrive.

Turtlesim (Publisher)
    ↓ sends position data continuously
    ↓
Topic: /turtle1/pose
    ↓ your subscriber listens here
    ↓
Your Code (Subscriber)
    ↓ (receives data every ~50 ms)
    ↓
Callback function fires → Prints position data

Key Difference: Callback Functions​

Publisher uses a timer:

self.timer = self.create_timer(0.5, self.timer_callback)
# timer_callback fires every 0.5 seconds (whether or not subscribers exist)

Subscriber uses a callback:

self.subscription = self.create_subscription(
    Pose, '/turtle1/pose', self.listener_callback, 10
)
# listener_callback fires ONLY when a message arrives

This is event-driven: your code waits for events (messages) instead of running on a schedule.

Layer 2: Creating Your First Subscriber Node​

Step 1: Create a File​

Create a new file called subscriber_node.py:

touch ~/subscriber_node.py

Step 2: Write the Subscriber Code​

Type this code carefully:

import rclpy
from rclpy.node import Node
from turtlesim.msg import Pose


class TurtlePoseReader(Node):
    def __init__(self):
        super().__init__('turtle_pose_reader')

        # Create a subscription to /turtle1/pose
        # When a message arrives, listener_callback is called
        self.subscription = self.create_subscription(
            Pose,                    # Message type
            '/turtle1/pose',         # Topic name
            self.listener_callback,  # Callback function
            10                       # Queue size
        )

        # Prevent unused variable warning
        self.subscription

        # For tracking
        self.counter = 0

    def listener_callback(self, msg):
        """Called whenever a Pose message is received"""

        # Extract position data from the message
        x = msg.x
        y = msg.y
        theta = msg.theta  # Rotation angle

        # Log the position
        self.get_logger().info(
            f'Turtle pose: x={x:.2f}, y={y:.2f}, theta={theta:.2f}'
        )

        self.counter += 1


def main(args=None):
    """Entry point for the ROS 2 node"""

    rclpy.init(args=args)
    node = TurtlePoseReader()

    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()

Step 3: Understand the Code​

Imports:

import rclpy
from rclpy.node import Node
from turtlesim.msg import Pose  # Message type for turtle position

Subscription Creation:

self.subscription = self.create_subscription(
    Pose,                   # Message type (from turtlesim package)
    '/turtle1/pose',        # Topic to subscribe to
    self.listener_callback, # Function to call when message arrives
    10                      # Queue size
)

Callback Function:

def listener_callback(self, msg):
    # This function is called EVERY TIME a message arrives
    x = msg.x        # Extract x coordinate
    y = msg.y        # Extract y coordinate
    theta = msg.theta  # Extract rotation
    self.get_logger().info(...)  # Log it

Key difference from publisher:

  • No timer! The callback fires only when messages arrive
  • The callback receives the message as a parameter: msg
  • You extract fields from the message: msg.x, msg.y, etc.

💬 AI CoLearning: Understanding Callbacks​

The callback pattern is important in ROS 2 (and all event-driven systems).

Ask Claude:

  1. "Why do we use callbacks instead of polling? Why not check if a message arrived in a while loop?"
  2. "What happens if my callback takes 1 second to process, but messages arrive every 50 ms?"
  3. "Can I have multiple callbacks on the same topic?"

Expected answers:

  1. Callbacks are more efficient—code only runs when something happens
  2. If callback is slow, messages queue up in the queue_size buffer
  3. No, but one node can have multiple subscriptions to different topics

This teaches you about event-driven architecture—the foundation of responsive systems.

🎓 Expert Insight: Pub/Sub Decoupling​

Here's the power of the pub/sub pattern:

Before ROS 2 (Direct Coupling)​

# ❌ BAD: Publisher and Subscriber tightly coupled
class Publisher:
    def send_data(self):
        subscriber.receive(data)  # Direct call!

# Publisher needs to know about Subscriber
# Tightly coupled, hard to test, hard to reuse

With ROS 2 (Loose Coupling)​

# ✅ GOOD: Publisher and Subscriber loosely coupled via topic
class Publisher(Node):
    def __init__(self):
        self.pub = self.create_publisher(Twist, '/cmd_vel', 10)

    def send_data(self):
        self.pub.publish(data)  # Publish to topic

class Subscriber(Node):
    def __init__(self):
        self.sub = self.create_subscription(Twist, '/cmd_vel', self.callback, 10)

    def callback(self, msg):
        # Process message

# Publisher doesn't know Subscriber exists
# Subscriber doesn't know Publisher exists
# They communicate only through the topic (/cmd_vel)

Benefits of loose coupling:

  1. Reusability: Publisher works with ANY subscriber
  2. Testing: Test publisher and subscriber independently
  3. Scalability: Add 10 subscribers to same topic—publisher doesn't change
  4. Flexibility: Replace publisher with different implementation—subscribers don't break

This is why ROS 2 is so powerful.

Experiment: Modify the Subscriber​

Now modify the subscriber to do something more interesting.

Experiment 1: Detect When Turtle Moves​

def listener_callback(self, msg):
    x = msg.x
    y = msg.y
    theta = msg.theta

    # Only log if turtle has moved significantly
    if self.counter > 0 and (x != self.prev_x or y != self.prev_y):
        self.get_logger().info(f'Turtle moved! New position: {x:.2f}, {y:.2f}')

    self.prev_x = x
    self.prev_y = y
    self.counter += 1

# In __init__:
self.prev_x = 0.0
self.prev_y = 0.0

Experiment 2: Calculate Velocity​

import time

def listener_callback(self, msg):
    current_time = time.time()

    if self.counter > 0:
        dt = current_time - self.prev_time
        dx = msg.x - self.prev_x
        dy = msg.y - self.prev_y

        vx = dx / dt  # Velocity in x direction
        vy = dy / dt  # Velocity in y direction

        self.get_logger().info(f'Velocity: vx={vx:.2f}, vy={vy:.2f}')

    self.prev_x = msg.x
    self.prev_y = msg.y
    self.prev_time = current_time
    self.counter += 1

# In __init__:
self.prev_x = 0.0
self.prev_y = 0.0
self.prev_time = time.time()

Experiment 3: Detect Rotation​

def listener_callback(self, msg):
    theta = msg.theta

    # Is the turtle rotating?
    if abs(msg.angular_velocity) > 0.1:  # Rotating faster than 0.1 rad/s
        self.get_logger().info(f'Turtle is rotating! Angle: {theta:.2f}')
    else:
        self.get_logger().info(f'Turtle is stationary. Angle: {theta:.2f}')

    self.counter += 1

Exercise: Pub + Sub Together​

Now you understand both publishers and subscribers. Let's combine them:

Task: Create a Feedback Loop​

Build a system where:

  1. Publisher sends "move forward" command every 0.5 seconds
  2. Subscriber reads turtle position every ~50 ms
  3. When turtle has moved 1 meter, stop

Here's a skeleton:

import rclpy
from rclpy.node import Node
from geometry_msgs.msg import Twist
from turtlesim.msg import Pose


class TurtleController(Node):
    def __init__(self):
        super().__init__('turtle_controller')

        # Publisher (sends commands)
        self.publisher_ = self.create_publisher(Twist, '/turtle1/cmd_vel', 10)

        # Subscriber (reads position)
        self.subscription = self.create_subscription(
            Pose, '/turtle1/pose', self.pose_callback, 10
        )

        # Timer (sends commands periodically)
        self.timer = self.create_timer(0.5, self.timer_callback)

        # Track starting position
        self.start_x = None
        self.distance_traveled = 0.0

    def pose_callback(self, msg):
        # Called when position data arrives
        if self.start_x is None:
            self.start_x = msg.x  # Record starting position

        # Calculate distance traveled
        self.distance_traveled = msg.x - self.start_x
        self.get_logger().info(f'Distance: {self.distance_traveled:.2f} meters')

        # Stop if we've moved 1 meter
        if self.distance_traveled >= 1.0:
            self.get_logger().info('Reached goal! Stopping.')
            rclpy.shutdown()

    def timer_callback(self):
        # Called every 0.5 seconds
        msg = Twist()
        msg.linear.x = 1.0  # Move forward
        self.publisher_.publish(msg)


def main(args=None):
    rclpy.init(args=args)
    node = TurtleController()

    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()

Run it:

python3 turtle_controller.py

Expected behavior:

  • Turtle starts moving forward
  • Node logs distance traveled
  • When distance reaches 1.0 meter, turtle stops
  • Node shuts down

What this demonstrates:

  • Publisher sends commands (control)
  • Subscriber receives feedback (sensing)
  • Node makes decisions based on feedback
  • This is the foundation of closed-loop control

Troubleshooting​

Error: "ModuleNotFoundError: No module named 'turtlesim'"​

Cause: turtlesim not installed or not in Python path

Solution:

sudo apt install ros-humble-turtlesim

Error: "Callback never fires, no messages"​

Diagnosis steps:

  1. Is Turtlesim running?

    ros2 node list | grep turtlesim

  2. Is the topic being published?

    ros2 topic echo /turtle1/pose

  3. Is your subscriber listening to the right topic?

    ros2 node info /turtle_pose_reader

    Check the "Subscribers" section.

If all checks pass, the callback should fire. If not, check:

  • Topic name spelling (case-sensitive!)
  • Message type matches (Pose from turtlesim)

Error: "Invalid message type: Pose"​

Cause: Import statement wrong

Solution:

from turtlesim.msg import Pose  # Correct
# NOT:
from std_msgs.msg import Pose  # Wrong

Pose is specific to turtlesim. Check the import!
