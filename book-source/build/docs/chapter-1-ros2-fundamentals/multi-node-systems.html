Building Multi-Node Systems

Introduction​

You now understand publishers and subscribers. In this final lesson, you'll combine them to build a complete robot system.

A real robot doesn't have one node. It has many:

  • Sensor reader nodes (read cameras, Lidar, etc.)
  • Processing nodes (detect obstacles, plan paths)
  • Decision nodes (decide what to do)
  • Motor driver nodes (execute commands)

All independent. All communicating via topics. All working together seamlessly.

What you'll build:

  • 3 nodes working as a system
  • Each node independent, doing one thing well
  • Coordinated through topics (not direct function calls)
  • Architecture you could scale to 10, 50, 100 nodes

System Architecture: Pen Drawer​

Let's build a system that makes the Turtlesim turtle draw a square.

Architecture Diagram​

┌──────────────────────────────┐
│   Movement Planner Node      │  (decides what to do)
│   publishes movement goals   │
└──────────────┬───────────────┘
               │ /movement/goal
               ↓
┌──────────────────────────────┐
│   Command Generator Node     │  (converts goals to commands)
│   reads goals, publishes cmd │
└──────────────┬───────────────┘
               │ /turtle1/cmd_vel
               ↓
┌──────────────────────────────┐
│   Turtlesim                  │  (executes commands)
│   reads commands, moves      │
└──────────────────────────────┘

Three nodes:

  1. Movement Planner — High-level: "move forward 1 meter"
  2. Command Generator — Low-level: "move forward at 1 m/s"
  3. Turtlesim — Hardware: moves the turtle

Each node is independent. Each node is reusable. They communicate via topics.

Node 1: Movement Planner​

This node decides what to do: move forward 1 meter, rotate 90 degrees, repeat 4 times to make a square.

File: planner_node.py

import rclpy
from rclpy.node import Node
from std_msgs.msg import String
import time


class MovementPlanner(Node):
    def __init__(self):
        super().__init__('movement_planner')

        # Publisher: sends high-level movement goals
        self.publisher_ = self.create_publisher(String, '/movement/goal', 10)

        # Timer: plan every 2 seconds (time for robot to complete previous goal)
        self.timer = self.create_timer(2.0, self.timer_callback)

        # Track which step we're on
        self.step = 0
        self.completed = False

    def timer_callback(self):
        if self.completed:
            return

        # Create a goal message
        msg = String()

        # Square pattern: move, rotate, move, rotate, move, rotate, move, rotate (stop)
        step_mod = self.step % 8

        if step_mod == 0:
            msg.data = 'move_forward'
        elif step_mod == 1:
            msg.data = 'rotate_right'
        elif step_mod == 2:
            msg.data = 'move_forward'
        elif step_mod == 3:
            msg.data = 'rotate_right'
        elif step_mod == 4:
            msg.data = 'move_forward'
        elif step_mod == 5:
            msg.data = 'rotate_right'
        elif step_mod == 6:
            msg.data = 'move_forward'
        elif step_mod == 7:
            msg.data = 'rotate_right'
            msg.data = 'stop'  # After 4 rotations, stop
            self.completed = True
            self.get_logger().info('Square complete!')

        self.publisher_.publish(msg)
        self.get_logger().info(f'Planning: {msg.data}')
        self.step += 1


def main(args=None):
    rclpy.init(args=args)
    node = MovementPlanner()

    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()

What this node does:

  • Publishes high-level goals ("move_forward", "rotate_right")
  • One goal every 2 seconds
  • Doesn't know HOW the robot moves, only WHAT it should do

Node 2: Command Generator​

This node translates high-level goals into low-level motor commands.

It subscribes to /movement/goal (from planner) and publishes to /turtle1/cmd_vel (for Turtlesim).

File: command_generator_node.py

import rclpy
from rclpy.node import Node
from std_msgs.msg import String
from geometry_msgs.msg import Twist


class CommandGenerator(Node):
    def __init__(self):
        super().__init__('command_generator')

        # Subscribe to movement goals
        self.subscription = self.create_subscription(
            String, '/movement/goal', self.goal_callback, 10
        )

        # Publish motor commands
        self.publisher_ = self.create_publisher(Twist, '/turtle1/cmd_vel', 10)

    def goal_callback(self, msg):
        """Convert high-level goal to low-level command"""

        # Create a Twist command
        cmd = Twist()

        if msg.data == 'move_forward':
            cmd.linear.x = 1.0   # Move forward at 1 m/s
            cmd.angular.z = 0.0  # Don't rotate
            self.get_logger().info('Generating command: move forward')

        elif msg.data == 'rotate_right':
            cmd.linear.x = 0.0    # Don't move forward
            cmd.angular.z = -0.5  # Rotate clockwise at 0.5 rad/s
            self.get_logger().info('Generating command: rotate right')

        elif msg.data == 'stop':
            cmd.linear.x = 0.0
            cmd.angular.z = 0.0
            self.get_logger().info('Generating command: stop')

        # Publish the command
        self.publisher_.publish(cmd)


def main(args=None):
    rclpy.init(args=args)
    node = CommandGenerator()

    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()

What this node does:

  • Reads high-level goals from planner
  • Converts them to motor commands (Twist)
  • Publishes commands for Turtlesim
  • Doesn't know what goals are coming, just handles them

💬 AI CoLearning: System Design​

Now you've built a 3-node system. Let's think about architecture.

Ask Claude:

  1. "Why did we separate the Planner from the Command Generator? Why not one node?"
  2. "If I wanted to add a node that logs every movement goal, where would I plug it in?"
  3. "Could I replace Turtlesim with a real robot without changing the other two nodes?"

Expected answers:

  1. Separation of concerns: high-level planning vs low-level control. Reusable.
  2. Subscribe to /movement/goal topic. No changes needed to planner or generator.
  3. Yes! As long as the new robot subscribes to /turtle1/cmd_vel, the planner and generator still work.

This teaches architectural thinking: how to design systems that scale and adapt.

Exercise: Add a Monitoring Node​

Let's add a fourth node that monitors the system without changing the existing three.

File: monitor_node.py

import rclpy
from rclpy.node import Node
from std_msgs.msg import String
from geometry_msgs.msg import Twist


class SystemMonitor(Node):
    def __init__(self):
        super().__init__('system_monitor')

        # Subscribe to goals
        self.goal_sub = self.create_subscription(
            String, '/movement/goal', self.goal_callback, 10
        )

        # Subscribe to commands
        self.cmd_sub = self.create_subscription(
            Twist, '/turtle1/cmd_vel', self.cmd_callback, 10
        )

        self.goal_count = 0
        self.cmd_count = 0

    def goal_callback(self, msg):
        self.goal_count += 1
        self.get_logger().info(
            f'[MONITOR] Goal #{self.goal_count}: {msg.data}'
        )

    def cmd_callback(self, msg):
        self.cmd_count += 1
        self.get_logger().info(
            f'[MONITOR] Command #{self.cmd_count}: '
            f'v={msg.linear.x:.1f}, w={msg.angular.z:.1f}'
        )


def main(args=None):
    rclpy.init(args=args)
    node = SystemMonitor()

    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()

Run it in a 4th terminal:

python3 ~/monitor_node.py

Output:

[MONITOR] Goal #1: move_forward
[MONITOR] Command #1: v=1.0, w=0.0
[MONITOR] Goal #2: rotate_right
[MONITOR] Command #2: v=0.0, w=-0.5
...

Key insight: You added a monitoring node without changing any existing nodes. This is loose coupling!

Understanding Decoupling​

Let's see why loose coupling matters.

Scenario: Want to Add a Path Follower?​

In traditional single-process code:

def draw_square():
    # All in one function
    move_forward()
    rotate_right()
    move_forward()
    rotate_right()
    # Hard to modify, hard to test

To add a path follower, you'd need to rewrite the whole thing.

With ROS 2:

Just add a new node that:
- Subscribes to /movement/goal
- Does whatever you want
- No changes to existing nodes!

Example: Add a visualization node that shows the planned path:

class PathVisualizer(Node):
    def __init__(self):
        super().__init__('path_visualizer')
        self.subscription = self.create_subscription(
            String, '/movement/goal', self.callback, 10
        )

    def callback(self, msg):
        if msg.data == 'move_forward':
            print('▶️  Moving forward')
        elif msg.data == 'rotate_right':
            print('↻  Rotating right')

# Run it. It works without any changes to existing nodes!

Troubleshooting Multi-Node Systems​

Issue: "Nodes not communicating"​

Diagnosis:

ros2 topic list                    # Are topics created?
ros2 node list                     # Are nodes running?
ros2 node info /movement_planner   # What does planner publish?
ros2 node info /command_generator  # What does generator subscribe to?
ros2 topic echo /movement/goal     # Are messages flowing?

Common causes:

  • Typo in topic name (case-sensitive!)
  • Wrong message type
  • Node crashed (check terminal for errors)

Issue: "Turtle doesn't move even though messages are publishing"​

Check:

  1. Is Turtlesim running? ros2 node list | grep turtlesim
  2. Are commands published to /turtle1/cmd_vel? ros2 topic echo /turtle1/cmd_vel
  3. Are message values sensible? (linear.x should be ~1.0, not 0.0)

If messages exist and have good values, check the Twist message type matches.

Issue: "Multiple nodes interfering with each other"​

Diagnosis:

ros2 node info /command_generator  # Multiple subscribers?
ros2 topic info /turtle1/cmd_vel   # Multiple publishers?

If two nodes publish to same topic, the last message wins (undefined behavior).

Solution: Each node should publish to different topics OR coordinate via a single topic with well-defined messages.

Capstone Challenge: Build a Complete System​

Here's your final challenge for Chapter 1.

Requirements​

Design and implement a 4-node system where:

  1. Node A (Sensor Simulator): Publishes fake sensor data (obstacle distance, 0-10 meters)

    Topic: /sensor/distance
    Message: Float32 (distance in meters)

  2. Node B (Obstacle Detector): Reads sensor data, detects if too close (< 1 meter)

    Subscribe: /sensor/distance
    Publish: /obstacle/detected (String: "CLEAR" or "OBSTACLE")

  3. Node C (Planner): Reads obstacles, decides action

    Subscribe: /obstacle/detected
    Publish: /movement/goal (String: "move_forward" or "stop")

  4. Node D (Motor Driver): Reads goals, sends motor commands to Turtlesim

    Subscribe: /movement/goal
    Publish: /turtle1/cmd_vel

Your Task​

  1. Write Node A (sensor simulator)
  2. Write Node B (obstacle detector)
  3. Write Node C (planner)
  4. Write Node D (motor driver) - can reuse from earlier lessons
  5. Run all 4 together: Turtlesim + 4 nodes = 5 nodes total
  6. Verify: Turtle should move forward, stop when obstacle detected, move again when clear

Success Criteria​

  • All 4 nodes run without errors
  • ros2 node list shows all 4
  • Topics connect correctly (verified with ros2 topic echo)
  • Turtle behavior demonstrates obstacle avoidance
  • No direct function calls between nodes (all via topics)
  • Can add a 5th monitoring node without breaking existing 4
